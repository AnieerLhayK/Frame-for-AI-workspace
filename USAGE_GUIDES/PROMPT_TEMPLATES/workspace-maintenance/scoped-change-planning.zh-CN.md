# 限定范围变更规划

当维护请求可能涉及多个拥有层时使用此 prompt。

```text
任务：
为此工作区维护请求规划最小的安全变更面：
<粘贴请求>

开始：
1. 运行 `git status --short --branch --untracked-files=all`。
2. 如果任务 id 不明确，运行 `workspace task list`。
3. 解析最佳候选任务 id。
4. 如果多个拥有层仍有可能，运行：
   `workspace changes plan <task-id> --intent <intent> --goal "<goal>"`

防护措施：
- 不要仅仅为了让范围验证通过而扩大任务。
- 当一个请求涉及多个拥有层时，优先拆分为多个限定范围的提交。
- 除非用户明确批准清理，否则不要移动、删除或清理被忽略的文件。
- 将 prompt 复用视为指导性建议，它不授予写权限。
- 保留用户变更，除非明确要求否则不要提交。

预期输出：
- 推荐的任务 id 及原因。
- 候选变更文件组。
- 应避免的文件组。
- 验证命令。
- 任务是否应拆分为多个提交。
```
