# 上下文目录结构重构

当 workspace 的记忆目录或工具目录积累了散落的登记表、报告、适配器和历史
记录时，使用以下方法。

## 先设计 seam

根目录保留小而稳定的接口，把实现细节隐藏在职责目录中。优先使用一个深入
的加载器或 CLI 接口，避免大量调用方理解文件布局。canonical 源负责事实和
行为；生成投影与旧入口是适配器，不是独立源。

## 迁移规则

1. 移动前盘点脚本、测试、manifest、发布器、活动文档和人工流程中的消费者。
2. 分离活动源、生成快照和历史证据。
3. 只有在读写器都切换到新路径后才移动追加型事实；禁止维护两个可写副本。
4. 存在直接消费者时，用生成的兼容投影保留旧机器可读形状。
5. 记录旧到新路径别名，不改写历史记录原文。
6. 提交前验证 canonical 源、投影一致性、引用和公共发布边界。

## AI 导航

提供一个轻量机器索引，记录读取条件和路径别名。根 README 负责人工导航，
实时摘要继续通过现有 workspace CLI 提供。历史内容应可定向读取，但不进入
普通启动上下文。

## 可复用验证

先运行加载器和适配器的聚焦测试，再运行结构验证器、全部脚本测试、报告新鲜度、
health 和 `git diff --check`。结案前把解析后的变更范围与任务记录比较。这套
顺序同时适用于 scripts 职责重构和 PROJECT_CONTEXT 目录迁移。
