# Runtime Loop Patterns

Runtime loop records turn user-visible drift into auditable evolution.

## Diagnosis Packet

Use when runtime behavior fails or drifts. It should identify source skill, task type, failed layer, evidence, severity, suggested patch scope, and next owner.

## Handoff Packet

Use when diagnosis should move to a maintainer. It should name the diagnosis, target maintainer, recommended files, patch type, risk, and acceptance criteria.

## Patch Note

Use when a maintainer accepts, rejects, or defers a change. It should explain files changed, expected effect, risk, rollback plan, and whether generator generalization is justified.

## Validation Note

Use after a patch. It should record before/after behavior, test prompt, observed improvement, remaining issue, pass/fail, reviewer, and next action.

## Generalization Note

Use only when a patch may contain a reusable generator or shared-protocol lesson. It should include the lesson, reason, target, risk, and decision.

## Ledger

Ledgers are indexes, not full stories. They make IDs discoverable and connect diagnosis, patch, validation, and generalization records.

## Runtime Audit Trail

The useful trail is:

```text
runtime failure -> diagnosis -> handoff -> maintainer decision -> patch -> validation -> optional generalization -> ledger update
```

## Status Decisions

- Accepted: the owner agrees the patch or lesson should proceed.
- Rejected: the change is not appropriate, and the reason should be recorded.
- Deferred: the issue is real but blocked, low priority, or waiting for more evidence.

## Why Durable Records Matter

Runtime failures often recur. Durable records prevent future maintainers from rediscovering the same cause without context.

## Why The Loop Should Not Be Fully Automatic

Some runtime failures are style-specific, user-specific, or character-specific. Human or maintainer judgment prevents local fixes from becoming broad regressions.

## Patterns From Experience

### Generalization Note As Decision Record (GEN-20260613-001)

The first completed generalization note (ZYC evidence-aware interaction scaffold) revealed an important pattern:

The generalization note format works for both **proposed** (backlog = true) and **completed** (backlog = false, accepted = true) decisions. The key fields:

- `generalizable: true` — is this lesson reusable outside its source character?
- `decision.backlog: false` — this was implemented, not just filed for later.
- `risk_if_generalized` — documenting what NOT to promote (ZYC-specific values, imagery, relationship advice).

This means the generalization backlog directory contains both pending proposals and completed records. They're distinguished by `decision.backlog`, not by directory or filename.

### Manual ID Generation Is A Known Gap

Currently, all runtime loop IDs (DIAG-, HANDOFF-, PATCH-, VAL-, GEN-) and ledger rows are created manually. This works at low volume but will become error-prone under real drift frequency. The ledger (95 lines for ~30 entries) is still manageable, but the ID pattern (YYYYMMDD-NNN with source character suffix) should be automated before the first week of real runtime failures.

### Template-First Packet Creation Works

The initial runtime loop design created reusable templates (`packages/character-system/shared/templates/diagnosis_packet.md`, `handoff_packet.md`, etc.) before any real instance. When the first generalization note was needed, the template guided its structure without requiring redesign. This validates the "template-first" approach for runtime loop infrastructure.

### Ledger Update After Every Loop Completion

Each completed loop phase should append a row to the corresponding ledger. The current ledger format (Markdown table with ID, date, source, lesson, decision, target, status) is lightweight enough to maintain manually but should be generated by script before scaling.
